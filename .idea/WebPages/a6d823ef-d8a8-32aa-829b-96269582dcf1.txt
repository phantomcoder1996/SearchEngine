           <p>First, a puzzle: What does the following code print?</p> 
           <pre><code>public class RecursiveStatic {
    public static void main(String[] args) {
        System.out.println(scale(5));
    }

    private static final long X = scale(10);

    private static long scale(long value) {
        return X * value;
    }
}
</code></pre> 
           <p>Answer: </p> 
           <blockquote class="spoiler"> 
            <p> 0</p> 
           </blockquote> 
           <p>Spoilers below.</p> 
           <hr> 
           <p>If you print <code>X</code> in scale(long) and redefine <code>X = scale(10) + 3</code>, the prints will be <code>X = 0</code> then <code>X = 3</code>. This means that <code>X</code> is temporarily set to <code>0</code> and later set to <code>3</code>. This is a violation of <code>final</code>!</p> 
           <blockquote> 
            <p>The static modifier, in combination with the final modifier, is also used to define constants. The final modifier indicates that the value of <strong>this field cannot change</strong>.</p> 
           </blockquote> 
           <p>Source: <a href="https://docs.oracle.com/javase/tutorial/java/javaOO/classvars.html" rel="noreferrer">https://docs.oracle.com/javase/tutorial/java/javaOO/classvars.html</a> [emphasis added]</p> 
           <hr> 
           <p>My question: Is this a bug? Is <code>final</code> ill-defined?</p> 
           <hr> 
           <p>Here is the code that I am interested in. <code>X</code> will take two different values: <code>0</code> and <code>3</code>. I believe this to be a violation of <code>final</code>.</p> 
           <pre><code>public class RecursiveStatic {
    public static void main(String[] args) {
        System.out.println(scale(5));
    }

    private static final long X = scale(10) + 3;

    private static long scale(long value) {
        System.out.println("X = " + X);
        return X * value;
    }
}
</code></pre>

           <div class="post-text" itemprop="text"> 
            <p>A very interesting find. To understand it we need to dig into the Java Language Specification (<a href="https://docs.oracle.com/javase/specs/jls/se9/html" rel="noreferrer">JLS</a>).</p> 
            <p>The reason is that <code>final</code> only allows one <em>assignment</em>. The default value, however, is no <em>assignment</em>. In fact, every <em>such variable</em> (class variable, instance variable, array component) points to its <em>default value</em> from the beginning, before <em>assignments</em>. The first assignment then changes the reference.</p> 
            <hr> 
            <h2>Class variables and default value</h2> 
            <p>Take a look at the following example:</p> 
            <pre><code>private static Object x;

public static void main(String[] args) {
    System.out.println(x); // Prints 'null'
}
</code></pre> 
            <p>We did not explicitly assign a value to <code>x</code>, though it points to <code>null</code>, it's default value. Compare that to <a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-4.html#jls-4.12.5" rel="noreferrer">§4.12.5</a>:</p> 
            <blockquote> 
             <p>Initial Values of Variables</p> 
             <p>Each <strong>class variable</strong>, instance variable, or array component is initialized with a <strong>default value</strong> when it is <strong>created</strong> (<a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-15.html#jls-15.9" rel="noreferrer">§15.9</a>, <a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-15.html#jls-15.10.2" rel="noreferrer">§15.10.2</a>)</p> 
            </blockquote> 
            <p>Note that this only holds for those kind of variables, like in our example. It does not hold for local variables, see the following example:</p> 
            <pre><code>public static void main(String[] args) {
    Object x;
    System.out.println(x);
    // Compile-time error:
    // variable x might not have been initialized
}
</code></pre> 
            <p>From the same JLS paragraph:</p> 
            <blockquote> 
             <p>A <strong>local variable</strong> (<a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-14.html#jls-14.4" rel="noreferrer">§14.4</a>, <a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-14.html#jls-14.14" rel="noreferrer">§14.14</a>) must be <strong>explicitly given a value</strong> before it is used, by either initialization (<a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-14.html#jls-14.4" rel="noreferrer">§14.4</a>) or assignment (<a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-15.html#jls-15.26" rel="noreferrer">§15.26</a>), in a way that can be verified using the rules for definite assignment (<a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-16.html" rel="noreferrer">§16 (Definite Assignment)</a>).</p> 
            </blockquote> 
            <hr> 
            <h2>Final variables</h2> 
            <p>Now we take a look at <code>final</code>, from <a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-4.html#jls-4.12.4" rel="noreferrer">§4.12.4</a>:</p> 
            <blockquote> 
             <p><em>final</em> Variables</p> 
             <p>A variable can be declared <em>final</em>. A <em>final</em> variable may only be <strong>assigned to once</strong>. It is a compile-time error if a <em>final</em> variable is assigned to unless it is <strong>definitely unassigned immediately prior to the assignment</strong> (<a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-16.html" rel="noreferrer">§16 (Definite Assignment)</a>).</p> 
            </blockquote> 
            <hr> 
            <h2>Explanation</h2> 
            <p>Now coming back to the your example, slightly modified:</p> 
            <pre><code>public static void main(String[] args) {
    System.out.println("After: " + X);
}

private static final long X = assign();

private static long assign() {
    // Access the value before first assignment
    System.out.println("Before: " + X);

    return X + 1;
}
</code></pre> 
            <p>It outputs</p> 
            <pre><code>Before: 0
After: 1
</code></pre> 
            <p>Recall what we have learned. Inside the method <code>assign</code> the variable <code>X</code> was <strong>not assigned</strong> a value to yet. Therefore, it points to its default value since it is an <strong>class variable</strong> and according to the JLS those variables always immediately point to their default values (in contrast to local variables). After the <code>assign</code> method the variable <code>X</code> is assigned the value <code>1</code> and because of <code>final</code> we can't change it anymore. So the following would not work due to <code>final</code>:</p> 
            <pre><code>private static long assign() {
    // Assign X
    X = 1;

    // Second assign after method will crash
    return X + 1;
}
</code></pre> 
            <hr> 
            <h2>Example in the JLS</h2> 
            <p>Thanks to @Andrew I found a JLS paragraph that covers exactly this scenario, it also demonstrates it.</p> 
            <p>But first let's take a look at</p> 
            <pre><code>private static final long X = X + 1;
// Compile-time error:
// self-reference in initializer
</code></pre> 
            <p>Why is this not allowed, whereas the access from the method is? Take a look at <a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-8.html#jls-8.3.3" rel="noreferrer">§8.3.3</a> which talks about when accesses to fields are restricted if the field was not initialized yet.</p> 
            <p>It lists some rules relevant for class variables:</p> 
            <blockquote> 
             <p>For a reference by simple name to a class variable <code>f</code> declared in class or interface <code>C</code>, it is a <strong>compile-time error if</strong>:</p> 
             <ul> 
              <li><p>The reference appears either in a class variable initializer of <code>C</code> or in a static initializer of <code>C</code> (<a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-8.html#jls-8.7" rel="noreferrer">§8.7</a>); and</p></li> 
              <li><p>The reference appears either in the initializer of <code>f</code>'s own declarator or at a point to the left of <code>f</code>'s declarator; and</p></li> 
              <li><p>The reference is not on the left hand side of an assignment expression (<a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-15.html#jls-15.26" rel="noreferrer">§15.26</a>); and</p></li> 
              <li><p>The innermost class or interface enclosing the reference is <code>C</code>.</p></li> 
             </ul> 
            </blockquote> 
            <p>It's simple, the <code>X = X + 1</code> is caught by those rules, the method access not. They even list this scenario and give an example:</p> 
            <blockquote> 
             <p>Accesses by methods are not checked in this way, so: </p> 
             <pre><code>class Z {
    static int peek() { return j; }
    static int i = peek();
    static int j = 1;
}
class Test {
    public static void main(String[] args) {
        System.out.println(Z.i);
    }
}
</code></pre> 
             <p>produces the output:</p> 
             <pre><code>0
</code></pre> 
             <p>because the variable initializer for <code>i</code> uses the class method peek to access the value of the variable <code>j</code> before <code>j</code> has been initialized by its variable initializer, at which point it <strong>still has its default value</strong> (<a href="https://docs.oracle.com/javase/specs/jls/se9/html/jls-4.html#jls-4.12.5" rel="noreferrer">§4.12.5</a>). </p> 
            </blockquote>

            <p><strong>Not a bug.</strong></p> 
            <p>When the first call to <code>scale</code> is called from</p> 
            <pre><code>private static final long X = scale(10);
</code></pre> 
            <p>It tries to evaluate <code>return X * value</code>. <code>X</code> has not been assigned a value yet and therefore the default value for a <code>long</code> is used (which is <code>0</code>).</p> 
            <p>So that line of code evaluates to <code>X * 10</code> i.e. <code>0 * 10</code> which is <code>0</code>.</p>

            <p>It's not a bug at all, simply put it is not an <em>illegal form</em> of forward references, nothing more. </p> 
            <pre><code>String x = y;
String y = "a"; // this will not compile 


String x = getIt(); // this will compile, but will be null
String y = "a";

public String getIt(){
    return y;
}
</code></pre> 
            <p>It's simply allowed by the Specification. </p> 
            <p>To take your example, this is exactly where this matches:</p> 
            <pre><code>private static final long X = scale(10) + 3;
</code></pre> 
            <p>You are doing a <em>forward reference</em> to <code>scale</code> that is not illegal in any way as said before, but allows you to get the default value of <code>X</code>. again, this is allowed by the Spec (to be more exact it is not prohibited), so it works just fine</p>

           <div class="post-text" itemprop="text"> 
            <p>Reading an uninitialized field of an object ought to result in a compilation error. Unfortunately for Java, it does not.</p> 
            <p>I think the fundamental reason why this is the case is "hidden" deep within the definition of how objects are instantiated and constructed, though I don't know the details of the standard.</p> 
            <p>In a sense, final is ill-defined because it doesn't even accomplish what its stated purpose is due to this problem. However, if all your classes are properly written, you don't have this problem. Meaning all fields are always set in all constructors and no object is ever created without calling one of its constructors. That seems natural until you have to use a serialization library.</p>

           <div class="post-text" itemprop="text"> 
            <p>Class level members can be initialized in code within the class definition. The compiled bytecode cannot initialize the class members inline. (Instance members are handled similarly, but this is not relevant for the question provided.)</p> 
            <p>When one writes something like the following:</p> 
            <pre><code>public class Demo1 {
    private static final long DemoLong1 = 1000;
}
</code></pre> 
            <p>The bytecode generated would be similar to the following:</p> 
            <pre><code>public class Demo2 {
    private static final long DemoLong2;

    static {
        DemoLong2 = 1000;
    }
}
</code></pre> 
            <p>The initialization code is placed within a static initializer which is run when the class loader first loads the class. With this knowledge, your original sample would be similar to the following:</p> 
            <pre><code>public class RecursiveStatic {
    private static final long X;

    private static long scale(long value) {
        return X * value;
    }

    static {
        X = scale(10);
    }

    public static void main(String[] args) {
        System.out.println(scale(5));
    }
}
</code></pre> 
            <ol> 
             <li>The JVM loads the RecursiveStatic as the jar's entry point.</li> 
             <li>The class loader runs the static initializer when the class definition is loaded.</li> 
             <li>The initializer calls the function <code>scale(10)</code> to assign the <code>static final</code> field <code>X</code>.</li> 
             <li>The <code>scale(long)</code> function runs while the class is partially initialized reading the uninitialized value of <code>X</code> which is the default of long or 0.</li> 
             <li>The value of <code>0 * 10</code> is assigned to <code>X</code> and the class loader completes.</li> 
             <li>The JVM runs the public static void main method calling <code>scale(5)</code> which multiplies 5 by the now initialized <code>X</code> value of 0 returning 0.</li> 
            </ol> 
            <p>The static final field <code>X</code> is only assigned once, preserving the guarantee held by the <code>final</code> keyword. For the subsequent query of adding 3 in the assignment, step 5 above becomes the evaluation of <code>0 * 10 + 3</code> which is the value <code>3</code> and the main method will print the result of <code>3 * 5</code> which is the value <code>15</code>.</p>
